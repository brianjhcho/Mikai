## Methodology (per O-020)

Two independent harnesses, both gated by the `--backend graphiti|local`
flag so Stage 6's LocalAdapter runs against the same bar:

| Harness | What it measures | Sample size | Scales |
|---|---|---|---|
| `scripts/eval_nodes.py` | Extraction quality | 10 random Entity nodes | accuracy 1-5, non-obviousness 1-5 |
| `scripts/eval_queries.py` | Retrieval groundedness | 10 starter queries × top-10 hits | relevance@10 (0-10), groundedness 1-5 |

The starter query set in `eval_queries.py::QUERIES` probes domains the
corpus is known to cover (poker, trading, MIKAI work, jobs, ideas).
Print and edit it for your own graph:

```bash
.venv/bin/python scripts/eval_queries.py --print-queries
```

## Run commands

```bash
cd infra/graphiti
.venv/bin/python ../../scripts/eval_nodes.py --n 10
.venv/bin/python ../../scripts/eval_queries.py --top-k 10
```

Each appends to `docs/evals/nodes-{YYYY-MM-DD}.md` and
`docs/evals/queries-{YYYY-MM-DD}.md` respectively. Re-runs on the same
day append rather than overwrite.

## Interpretation rubric

These are working thresholds, not externally validated. Use as a north
star, not a contract:

- **accuracy ≥ 4.0** — extraction faithfully represents source text
- **non-obviousness ≥ 3.0** — extraction adds signal beyond a keyword
  index (otherwise: why spend $0.005-$0.01/episode on DeepSeek?)
- **relevance@10 ≥ 6/10** — retrieval surfaces useful results more often
  than not
- **groundedness ≥ 4.0** — citations actually support the surfaced fact

If any dimension falls below threshold, the gap between current and
target is the prioritized work for the next iteration.

## Why this exists

O-020 is one of the blocking gates in `docs/OPEN.md`: until extraction
quality has a number, every downstream change ("better prompt", "Local
adapter wins", "L4 thread accuracy is up") is a vibe rather than a
measurement. Stage 6 (LocalAdapter) and Stage 5 (L4) both gate on
running the same harnesses against their own backends and meeting these
same thresholds.
